You are working in the Poster Stores project.

Goal:
Fix the mockup workflow so product mockups behave like real finalized mockup images, not inconsistent live previews.

Current problem:
The current mockup/gallery behavior is inconsistent:
1. Mockup placement/detection is still too dependent on x/y/width/height instead of a proper poster surface.
2. If AI detection fails, it applies a fallback area and overwrites or changes the current selected area. This is bad — detection must be non-destructive.
3. In the product image carousel/thumbnails, the poster is not visible inside the mockup until the mockup is selected. The thumbnail appears to show the bare mockup template/background.
4. In the lightbox/zoom view, the poster size changes and no longer matches the mockup frame. The lightbox appears to re-render the composite differently from the selected product image.

Important:
- Do not refactor the entire mockup system.
- Do not remove deterministic Sharp rendering.
- Do not make AI rendering the default.
- Do not create branches, PRs, commits, pushes, or GitHub automation.
- Keep changes targeted.
- Typecheck must pass.
- Existing manual templates must continue working.
- Existing public product gallery, hover image, primary image, and lightbox must continue working.

Main principle:
For public storefront views, use the final generated/flattened mockup image whenever it exists. Do not show a bare template background and do not recomposite differently in the thumbnail, main image, and lightbox.

Part 1 — Make AI placement detection non-destructive

Inspect the auto-placement detection flow in:
- mockupPlacementAnalyzer.ts
- analyze-placement endpoint
- MockupTemplateForm.tsx
- any placement fallback logic

Fix behavior:
- Pressing “Detect” or “Re-analyze” must never overwrite the current manual placement immediately.
- The AI result should be stored as a detected candidate.
- Admin must explicitly approve detected placement before it becomes active.
- If AI detection fails, keep the existing placement exactly as it was.
- If AI detection confidence is too low, keep existing placement exactly as it was and show “Needs review”.
- Do not write a fallback bounding box over an existing selection.
- Only create a fallback area if there is no existing placement at all, and even then store it only as a candidate/needs-review state, not active placement.

Expected behavior:
- Current manual area remains unchanged after failed detection.
- Current approved auto-detected placement remains unchanged after failed re-analysis.
- Failed detection stores detectedPlacementStatus = failed and detectedPlacementError, but does not change placementMode to a public-active mode.
