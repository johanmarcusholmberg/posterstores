STRIPE FLOW

Use this flow:

1. User completes checkout form.
2. Server creates order with status pending_payment.
3. Server creates Stripe Checkout Session for that order.
4. Frontend redirects user to Stripe Checkout.
5. Stripe redirects back to success or cancel page.
6. Server verifies payment through webhook or server-side session retrieval.
7. Order is updated to paid only after verified payment.
8. Admin can see the updated payment/order status.

CREATE CHECKOUT SESSION

Add an endpoint such as:

POST /api/orders/:id/create-checkout-session

Rules:
- Order must exist.
- Order must belong to requested storeKey.
- Order must have status pending_payment or draft.
- Order must not already be paid.
- Use server-side order/order_items snapshots to create Stripe line items.
- Do not trust frontend prices.
- Currency should come from order currency.
- Include order id and storeKey in Stripe metadata.
- Save stripe_checkout_session_id on the order.
- Return checkout URL to frontend.

Line items:
- Use poster title snapshot.
- Use size label snapshot.
- Use unit price snapshot.
- Use quantity.
- Use currency.
- Optional description: poster title + size.

Success URL:
APP_BASE_URL/order-confirmation/{orderId}?payment=success

Cancel URL:
APP_BASE_URL/checkout?payment=cancelled&orderId={orderId}

WEBHOOK

POST /api/stripe/webhook

Important:

- Do not parse body in a way that breaks signature verification.
- Handle at least:

  - checkout.session.expired
  - payment_intent.payment_failed if useful
- On checkout.session.completed:
  - Find order by stripe_checkout_session_id or metadata.orderId.
  - Confirm payment status.
  - Set order.status = paid.
  - Set order.payment_status = paid.
  - Set paid_at.
  - Save stripe_payment_intent_id if available.
- If session expires:
  - Set payment_status = cancelled or expired.
  - Keep order status as pending_payment or cancelled depending on chosen behavior.
