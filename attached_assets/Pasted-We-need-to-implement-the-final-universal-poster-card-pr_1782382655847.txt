We need to implement the final universal poster-card presentation system.

Do not create branches, PRs, commits, GitHub automation, or push. Make changes only and report back. Keep the implementation targeted and low-risk.

Context:
The current poster cards do not look good enough for a mixed poster catalogue. Landscape posters can look awkward, too small, or visually inconsistent compared with portrait posters. I want a cleaner Gallerix-like catalogue approach, but adapted to our store.

Important final UX decision:
Poster cards should be universal and fixed.

Do not add an admin selector for different poster-card presentation modes.

Do not create a “store cards” admin section.

Do not let stores choose between current / full-image / stage modes.

The poster-card layout, sizing, hover behavior, image behavior, and format handling should be fixed in code and consistent across all stores.

The only store-level customization I may want is the color of the actual card background, not the image/stage presentation.

Main goals:
1. Create one universal poster-card system for homepage and shop.
2. Show actual poster artwork by default.
3. Preserve the real format of portrait, landscape, and square posters.
4. Make landscape posters look like the same physical poster size rotated, not larger or more dominant than a regular 50x70 portrait poster.
5. Keep all product cards the same outer size within the same grid/section.
6. Keep all image/stage areas the same size within the same grid/section.
7. Use desktop-only hover mockup fade when a mockup exists.
8. Use highlight/shadow hover feedback on all cards.
9. Move NEW badge and heart/like button below the image in the regular shop grid.
10. Move the NEW badge below the image on the homepage as well, at least in sections where a poster-card style is used.
11. Avoid changing the product detail page unless a direct bug is found.

Universal card design:
All product cards must have identical outer dimensions within the same section/grid.

That means:
- Same card width.
- Same card image/stage area.
- Same text/meta area behavior.
- Same hover area.
- Same layout rules.

Only the artwork inside the stage should change size based on poster format.

Card image/stage behavior:
- Fixed image/stage area.
- Neutral/off-white stage background.
- Actual poster artwork centered inside the stage.
- Poster keeps its real aspect ratio.
- Poster has subtle border/shadow so it looks intentionally placed.
- Default poster artwork must not be cropped.
- Do not use object-cover for the default poster artwork.
- Do not use full-image/cover as the default poster-card presentation.

Important landscape rule:
Landscape posters should not take more visual space than a regular 50x70 portrait poster.

A landscape poster should:
- Appear wider and shorter.
- Feel like a 70x50 poster inside the same product tile.
- Not fill most of the card width if that makes it feel visually bigger than portrait posters.
- Not look like a hero image.
- Not dominate the grid.

Think of the Gallerix principle:
- Same product-card system.
- Different poster formats shown inside the same visual area.
- A landscape poster is the same physical poster size rotated, not a bigger product.

Suggested format-aware sizing inside the fixed stage:
These values are starting points and can be tuned visually.

Portrait poster:
- max-height around 82–88% of the stage.
- max-width around 62–72% of the stage.

Landscape poster:
- max-width around 62–72% of the stage.
- max-height around 42–52% of the stage.
- It must not look visually larger than a standard portrait poster.

Square poster:
- max-width around 62–72% of the stage.
- max-height around 62–72% of the stage.

Core principle:
Different proportions, same perceived physical size.

Homepage and shop consistency:
Homepage Featured posters, homepage New arrivals, and the shop grid should use the same base poster tile style.

They may differ in:
- number of cards per row
- section spacing
- hover intensity

They should not differ in:
- artwork presentation logic
- format handling
- default image behavior
- card anatomy
- title/price pattern
- basic visual language

Homepage Featured / New arrivals:
- Use the same base poster tile system as shop.
- Stronger hover feedback on desktop:
  - subtle lift, for example translateY(-2px).
  - stronger card shadow.
  - mockup fade if mockup exists.

Shop/store grid:
- Same base poster tile system.
- Calmer hover feedback on desktop:
  - shadow increase only.
  - no lift, or extremely minimal lift.
  - mockup fade if mockup exists.
  - if no mockup exists, shadow only.

Hover behavior:
- Hover-specific behavior is desktop only.
- Mobile should have no hover-specific image behavior.
- On mobile, tapping a card opens the poster/product page as usual.

Desktop hover with mockup:
- If a poster has a hover mockup/lifestyle image, fade from the actual poster artwork to the full-size mockup.
- The mockup should fill the full image/stage area.
- The mockup should use object-cover.
- The mockup should be an absolute overlay.
- Fade should be smooth in and out, around 600ms.
- Do not mount the hover image only on hover.
- Keep the hover image rendered/preloaded enough to avoid a first-hover snap.

Desktop hover without mockup:
- Do not swap the image.
- Do not zoom the image.
- The artwork should stay as-is.
- Apply only hover highlight:
  - shop grid: subtle shadow increase.
  - homepage cards: stronger shadow and subtle lift.

Card background color:
I do not want a full admin area for store cards.

I only want the possibility to control the color of the actual card background if this fits cleanly into the existing store/admin appearance settings.

Use or add a single store-level field only if needed, for example:
- productCardBackgroundColor

Meaning:
- Controls the card surface/background.
- Does not control poster sizing.
- Does not control poster layout.
- Does not control stage/full-image/current mode.
- Does not control hover behavior.
- Does not control shadow strength.
- Does not introduce a store-card layout system.

Place it in an existing store appearance/color area if possible. Do not create a new dedicated “store cards” admin section.

If a suitable existing color field already exists, reuse it instead of adding a new DB field.

If a new field is needed:
- Add it safely.
- Add/update DB schema.
- Add an idempotent startup migration.
- Add/update backend Zod/API schemas.
- Add/update frontend types.
- Do not overwrite existing store data.
- Use a sensible fallback color.
