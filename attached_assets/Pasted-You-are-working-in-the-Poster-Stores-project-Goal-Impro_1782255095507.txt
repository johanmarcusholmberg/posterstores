Part 9 — Sync only posters where the mockup is selected

Current problem:
When syncing mockups today, selecting a mockup template can generate that mockup for every poster in the gallery/store, even for posters where that mockup has not been selected.

This is wrong.

Desired rule:
A mockup sync/update must only render mockups for posters where that mockup/template is actually selected for that poster.

In other words:

* Active template = available to choose
* Selected mockup for poster = should be rendered for that poster
* Sync must use selected poster/mockup relationships, not all active templates for all posters

Correct behavior:

1. Poster-specific sync
   If the admin is editing one poster’s mockups:

* Sync should only update mockups selected for that poster.
* It must not apply the same template to other posters.

Example:
Poster A has templates 1 and 2 selected.
Poster B has template 3 selected.
If admin syncs Poster A, only Poster A + templates 1 and 2 should be rendered.

2. Selected mockup template sync
   If the admin selects a mockup template and runs sync:

* Only posters that already have that mockup/template selected should be updated.
* Do not create new poster_mockup rows for every poster.
* Do not apply the template to posters where it has not been selected.

Example:
Template 5 is selected for Poster A and Poster C only.
Syncing Template 5 should render Poster A + Template 5 and Poster C + Template 5 only.
It must not generate Template 5 for Poster B, D, E, etc.

3. Selected poster + selected mockup pairs
   If the UI allows selecting both posters and templates:

* Render only the intersection where the template is selected for that poster.
* If a template is selected in the UI but not assigned/selected for a poster, skip that poster/template pair and show a clear skipped reason.

Example:
Selected posters: A, B, C
Selected template: 5
Template 5 is selected only for A and C.
Sync result:

* A + 5 generated/updated
* C + 5 generated/updated
* B + 5 skipped: “Mockup not selected for this poster”

4. Bulk sync
   If bulk sync remains:

* It should mean “update all existing selected poster/mockup relationships”
* It should not mean “apply all active templates to all posters”
* Label it clearly:
  “Bulk update selected mockups”
  not “Sync all mockups”

If there is a separate action to apply templates to all posters, it must be a different explicit action:
“Apply selected template to all posters”
This action should require strong confirmation and should not be part of normal sync.

Server behavior:
Update mockupSync.ts so it builds work items from existing selected poster_mockup relationships, not from all posters × active templates.

Required server rules:

* Do not create poster_mockup rows for posters unless the mockup/template is explicitly selected for that poster.
* Do not use all published posters as the default target.
* Do not generate active templates for all posters automatically.
* Reject ambiguous sync requests that do not specify either:

  * poster IDs
  * template IDs
  * existing poster_mockup IDs
  * or explicit bulk update of existing selected rows
* If poster IDs and template IDs are supplied, filter to existing selected poster_mockup rows only.
* If a requested poster/template pair does not exist as a selected mockup row, skip it and report:
  “Mockup not selected for this poster.”

UI behavior:

* Rename “Run sync” if needed to make it clear:

  * “Update selected mockups”
  * “Sync this poster’s selected mockups”
  * “Bulk update selected mockups”
* Show planned count based on existing selected relationships:
  “5 selected poster/mockup pairs will be updated”
* Do not show a count based on all posters × all templates unless the action is explicitly applying templates to all posters.
* If nothing is selected for update, disable sync and show:
  “No selected mockups to sync.”

Manual QA:

* Template selected for one poster only → sync updates only that poster.
* Template selected for two posters only → sync updates only those two posters.
* Syncing one poster updates only mockups selected for that poster.
* Syncing one template does not create that mockup for unrelated posters.
* Active templates are not automatically applied to every poster.
* Bulk update updates existing selected rows only.
* Existing AI cost limits and general sync hard limits still apply.

3. Prefer auto-renumbering over hard errors:

   * Avoid duplicate values.

   * Save the resulting order as clean sort_order values.

6. If drag-and-drop is too much for this pass:
